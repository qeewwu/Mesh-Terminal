# Настройки узла (`/settings`)

`/settings` без аргументов показывает текущие значения. `/settings <параметр> <значение>`
меняет один параметр и сразу пишет его на устройство (`node.writeConfig(...)` в
`mesh_logger.py`). Это конфигурация **локального узла** — того, к которому подключён
`mesh_logger.py` по TCP, — а не удалённых узлов сети.

Список параметров — сознательно небольшой, курируемый (см. `SETTINGS_REGISTRY` в
`mesh_logger.py`), а не generic-доступ к произвольному protobuf-полю. Причины:

- **`network` (WiFi) не выставлена.** Соединение логгера с устройством идёт по тому же
  WiFi/TCP-каналу. Ошибка в SSID/пароле или отключение WiFi-интерфейса оборвала бы это же
  соединение — устройство стало бы недоступно удалённо.
- **`bluetooth`/`security` не выставлены.** Там PIN-коды и криптографические ключи; отдавать
  их на изменение через локальный unix-сокет без дополнительной защиты — лишний риск.
- **`display`/`power`/`audio`/остальные модули не выставлены** — не имеют смысла для
  headless-узла, подключённого только по WiFi.

Некоторые параметры (в первую очередь `lora.*`) применяются только после перезагрузки
устройства — сама по себе запись настройки её не вызывает. Для этого есть **`/reboot`**:
показывает предупреждение и требует явного `/reboot confirm` в течение 30 секунд (двухшаговое
подтверждение — случайно перезагрузить узел одной опечаткой не получится). После перезагрузки
TCP-соединение на секунду оборвётся, и `mesh_logger.py` переподключится сам (см. реконнект-логику
в `CLAUDE.md`) — сообщения, отправленные в этот момент, уйдут в outbox и досопустятся
автоматически.

## Параметры

| Параметр | Секция конфига | Тип | Что делает |
|---|---|---|---|
| `owner_long` | — (`setOwner`) | строка | Полное имя узла, как видят остальные участники сети |
| `owner_short` | — (`setOwner`) | строка | Короткое имя узла (обычно до 4 символов), показывается компактно в списках/логах |
| `role` | `device` | enum | Роль узла в сети — см. таблицу ролей ниже |
| `node_info_broadcast_secs` | `device` | целое, 60–86400 | Как часто узел рассылает свой NodeInfo (имя, аппаратные данные) остальным |
| `hop_limit` | `lora` | целое, 1–7 | Максимум ретрансляций (хопов) для пакетов, отправленных с этого узла |
| `tx_power` | `lora` | целое, 0–30 | Мощность передачи в дБм. `0` — не число ватт, а флаг «использовать значение по умолчанию для региона»: устройство само подставляет реальную мощность, и именно её (а не `0`) `/settings` покажет после перезагрузки — это ожидаемо, не баг |
| `position_broadcast_secs` | `position` | целое, 30–86400 | Как часто узел рассылает свою GPS-позицию |
| `position_smart_enabled` | `position` | булево | Если включено — позиция рассылается не по таймеру, а при значимом перемещении |
| `gps_mode` | `position` | enum: `ENABLED`, `DISABLED`, `NOT_PRESENT` | Режим встроенного GPS-приёмника |
| `fixed_position` | `position` | булево | Считать текущую позицию фиксированной (узел неподвижен, например базовая станция) |
| `telemetry_device_secs` | `telemetry` (модуль) | целое, 60–86400 | Как часто слать телеметрию самого устройства (батарея, напряжение, аптайм) |
| `telemetry_env_enabled` | `telemetry` (модуль) | булево | Слать ли телеметрию окружающей среды (температура/давление/влажность), если есть датчик |
| `mqtt_enabled` | `mqtt` (модуль) | булево | Включить мост LoRa ⇄ MQTT — сообщения уходят не только в сеть, но и в интернет (например, в [OneMesh](https://map.onemesh.ru)) |
| `mqtt_address` | `mqtt` (модуль) | строка | Адрес брокера, `host[:port]` (например `map.onemesh.ru:1883`) |
| `mqtt_username` | `mqtt` (модуль) | строка | Логин для подключения к брокеру |
| `mqtt_password` | `mqtt` (модуль) | строка | Пароль — записывается как есть, но `/settings` всегда показывает его маской (`••••••••`), никогда не в открытом виде |
| `mqtt_encryption_enabled` | `mqtt` (модуль) | булево | Шифровать пакеты в MQTT тем же PSK, что и канал (а не слать открытым текстом) |
| `mqtt_json_enabled` | `mqtt` (модуль) | булево | Дополнительно публиковать сообщения в виде JSON (удобно для сторонних интеграций) |
| `mqtt_tls_enabled` | `mqtt` (модуль) | булево | TLS-соединение с брокером — включайте, если брокер того требует |
| `mqtt_root` | `mqtt` (модуль) | строка | Корневой топик MQTT (например `msh/RU`) |
| `mqtt_uplink` | канал (`ChannelSettings`) | `<канал>:on\|off` | Ретранслировать канал из LoRa в MQTT — например `/settings mqtt_uplink Primary:on` |
| `mqtt_downlink` | канал (`ChannelSettings`) | `<канал>:on\|off` | Ретранслировать канал из MQTT обратно в LoRa — например `/settings mqtt_downlink Primary:off` |

`mqtt_uplink`/`mqtt_downlink` — единственные два параметра, которые пишут не в `moduleConfig`, а
в настройки конкретного канала (`node.writeChannel()`, а не `node.writeConfig()`), поэтому у них
свой формат значения — имя канала и флаг через двоеточие, а не просто число/булево/enum. В списке
`/settings` они показывают сводку по всем каналам сразу (`Primary:on, LongFast:off`), а не одно
значение — потому что состояние у каждого канала своё.

**Чтобы MQTT реально заработал**, обычно нужно выставить как минимум: `mqtt_enabled on`,
`mqtt_address`, при необходимости `mqtt_username`/`mqtt_password` (публичные брокеры вроде
mqtt.meshtastic.org работают без логина, у OneMesh и подобных сервисов — уточняйте у них), и
`mqtt_uplink <ваш_канал>:on` — без этого последнего шага сообщения продолжат ходить только по
LoRa, даже при включённом `mqtt_enabled`.

### Роли узла (`role`)

| Значение | Смысл |
|---|---|
| `CLIENT` | Обычный узел-участник сети (по умолчанию) |
| `CLIENT_MUTE` | Участвует в сети, но не ретранслирует чужие пакеты |
| `CLIENT_HIDDEN` | Не участвует в NodeInfo-обмене, минимальный "след" в сети |
| `CLIENT_BASE` | Клиент с фиксированной позицией (базовая станция без ретрансляции) |
| `ROUTER` | Инфраструктурный узел — ретранслирует пакеты, не показывается как обычный собеседник |
| `ROUTER_CLIENT` | Ретранслятор, который также участвует в чате как обычный узел |
| `ROUTER_LATE` | Ретранслятор с задержкой пересылки (снижает коллизии в плотных сетях) |
| `REPEATER` | Чистый повторитель на физическом уровне, без своего NodeInfo |
| `TRACKER` / `TAK_TRACKER` | Узел-трекер (частая отправка позиции), в т.ч. интеграция с TAK |
| `SENSOR` | Узел с датчиками, ориентирован на телеметрию, а не на чат |
| `TAK` | Интеграция с ATAK/WinTAK |
| `LOST_AND_FOUND` | Режим "потерялся/нашёлся" — агрессивная отправка позиции |

Смена `role` меняет поведение узла в сети (ретрансляция, видимость), но не трогает
WiFi/TCP — соединение логгера не пострадает.

## Формат значений

- **Целые** — просто число: `/settings hop_limit 4`
- **Булевы** — `on`/`off`, `true`/`false`, `1`/`0`, `да`/`нет`: `/settings position_smart_enabled on`
- **Enum** — имя значения без учёта регистра: `/settings role router`, `/settings gps_mode disabled`
- **Строки** (`owner_long`/`owner_short`, `mqtt_address`/`mqtt_username`/`mqtt_password`/`mqtt_root`) —
  как есть, без кавычек: `/settings owner_long Мой Узел`
- **Канал:флаг** (только `mqtt_uplink`/`mqtt_downlink`) — `<имя канала>:on|off`:
  `/settings mqtt_uplink Primary:on`

Диапазоны и допустимые enum-значения проверяются на стороне логгера до отправки на
устройство — при ошибке `/settings` вернёт понятное сообщение вместо того, чтобы отправить
на устройство мусор.

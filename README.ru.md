# Mesh-Terminal

Терминальный чат-клиент для mesh-радиосетей [Meshtastic](https://meshtastic.org/) — по WiFi, USB или Bluetooth LE.

![Python 3.10+](https://img.shields.io/badge/python-3.10%2B-blue)

*(English version: [README.md](README.md))*

> **Заметка:** по умолчанию интерфейс клиента на русском. Чтобы переключить на английский, укажите `MESH_LANG=en` в `.env` (см. [Конфигурацию](#конфигурация)).

## О проекте

Mesh ведёт постоянную, доступную для поиска историю переписки в сети и даёт быстрый терминальный интерфейс для чтения и отправки — каналы, личные сообщения, настоящие Meshtastic-ответы с цитатой, реакции, диагностика узлов и удалённая настройка устройства — всё с клавиатуры.

Устроен как два небольших независимых процесса вместо одного:

- **`mesh_logger.py`** — фоновый демон, который держит соединение с Meshtastic-устройством, пишет каждое сообщение в лог по дням и продолжает работать вне зависимости от того, открыт ли у вас терминал.
- **`mesh_chat.py`** — интерактивный клиент, в который вы реально печатаете. К устройству напрямую не обращается — говорит с логгером через локальный сокет, поэтому его можно запускать и закрывать сколько угодно раз: закрыли ноутбук, открыли завтра — история никуда не делась.

```
                    связь с устройством      Unix-сокет
Meshtastic ◄─────────────── mesh_logger.py ◄─────────── mesh_chat.py
 устройство      (WiFi/USB/BLE)     │                    (можно запускать
                                    ▼                     и останавливать
                                logs/*.log                 сколько угодно раз)
```

Такое разделение существует потому, что Meshtastic-устройство держит только **одно подключение одновременно**, независимо от транспорта. Выделив под это подключение один постоянно работающий процесс, интерактивный клиент может свободно перезапускаться, падать или ещё не быть запущенным — ничего не теряется, а несколько терминальных сессий могут одновременно смотреть один и тот же живой поток.

## Возможности

- **Чат** — все каналы сразу или фильтр на один; личные сообщения; настоящие Meshtastic-ответы с цитатой (`/reply`, видны и в официальных клиентах); тапбэк-реакции (`/react`)
- **История и поиск** — логи по дням автоматически, полнотекстовый поиск, история конкретного отправителя, статистика сообщений
- **Диагностика сети** — список узлов с батареей/SNR/хопами/статусом, позиция и расстояние/азимут до узла, traceroute, пинг (время отклика)
- **Управление устройством** — чтение и изменение настроек своей ноды удалённо, перезагрузка, переключение бота автоответчика
- **Надёжность** — сообщения, отправленные при офлайн-устройстве, встают в очередь и уходят при переподключении; автопереподключение и логгера к устройству, и клиента к логгеру; подтверждение доставки ✓/✗
- **Удобство** — имена узлов автоматически резолвятся через публичную карту [OneMesh](https://map.onemesh.ru/), мьют шумных отправителей, Tab-автодополнение команд/имён/каналов, язык интерфейса переключается между русским и английским (`MESH_LANG`)

## Чат

```
Primary [14:02:11] Вася Пупкин (VP): всем привет | 2
  ┆ [14:02:11] Вася Пупкин (VP): всем привет
Primary [14:02:34] 👍 Коля → [14:02:11] Вася Пупкин: «всем привет»
Primary [14:05:02] DM Петя (PT): встречаемся в 18:00 | 1
```

Без флагов показываются сразу все каналы, с пометкой имени каждого; отправка уходит в канал Primary. С флагом `--channel <имя>` (например, `--channel ping`) сессия сужается до одного канала — префикс пропадает, раз он и так понятен, и отправка тоже уходит именно в него.

Ответы держат переписку в виде треда:

```
> /reply
─── На что можно ответить (#1 — самое свежее) ───
  #1 [14:05] Петя: встречаемся в 18:00
  #2 [14:02] Вася Пупкин: всем привет

> /reply #1 хорошо, буду
Отвечаю на: [14:05] Петя — встречаемся в 18:00
```

Маршрут ответа определяется тем, на что вы отвечаете — на входящий DM уходит DM, на сообщение в канал — ответ в тот же канал, — так что думать об этом не нужно. Отвечать можно только на сообщения, полученные после запуска клиента (в истории на диске нет packet id Meshtastic).

## Узлы и диагностика сети

```
> /nodes
─── Видимые узлы ───
  Вася Пупкин (VP) 🔋82% SNR:6.5 · 2 хопа · Онлайн
  Петя (PT) 🔋54% SNR:-3.2 · 1 хоп · 12м назад

> /trace Вася
Трассирую до Вася Пупкин — может занять до 45с...
Маршрут туда: Я (ME) (?dB) → Relay (RL) (+5.0dB) → Вася Пупкин (VP) (+3.2dB)
Маршрут обратно: Вася Пупкин (VP) (?dB) → Relay (RL) (+2.8dB) → Я (ME) (+4.1dB)

> /pos Вася
📍 Вася Пупкин (VP): 55.75800, 37.61730, 180 м
   от меня: 4.2 км, азимут 132° (ЮВ)

> /ping Вася
🏓 Пингую Вася Пупкин (VP)...
🏓 Вася Пупкин (VP): доставлено за 2.3с, 2 хопа
```

`/nodes [online|names|hops]` сортирует по нужному ключу, остальные два работают как тайбрейкеры.

## История, поиск и статистика

```
> /search разбудите
─── Поиск: «разбудите» ───
ping [18:44:02] Я (ME): разбудите меня если что | 0

> /stats
─── Статистика ───
  Всего сообщений: 8349 за 13 дн.
  Самый активный час: 18:00 (612 сообщений)
```

`/search` ищет по всей истории на диске (с учётом текущего фильтра канала, если он задан); `/last <имя>` — последние сообщения одного отправителя; `/stats [день|узел]` — разбивка по дням или по самым активным.

## Настройки устройства

```
> /settings
─── Настройки узла ───
  role = CLIENT  Роль узла в сети (CLIENT, ROUTER, REPEATER, ...)
  hop_limit = 3  Максимум хопов для пакетов, отправленных с этого узла
  position_broadcast_secs = 900  Как часто рассылать позицию, секунды

> /settings hop_limit 4
✓ hop_limit = 4 (некоторые параметры требуют /reboot, чтобы вступить в силу)
```

Сознательно отобранный набор безопасных для изменения параметров — данные WiFi, PIN-коды Bluetooth и ключи безопасности намеренно исключены. Полный справочник параметров — в [`SETTINGS.ru.md`](SETTINGS.ru.md). `/reboot` (с обязательным `/reboot confirm` в течение окна времени) применяет настройки, которые вступают в силу только после перезагрузки.

## Надёжность

```
> привет из офлайна
⏳ В очереди: «привет из офлайна» — устройство офлайн, отправлю сразу при переподключении
```

Если устройство пропадает — перезагружается, моргнул WiFi — ничего из отправленного не теряется: сообщение встаёт в очередь и уходит сразу, как только связь восстановится. Сам клиент точно так же переподключается к логгеру, если тот перезапускается, и дорисовывает всё, что было пропущено.

Переподключение к устройству обычно автоматическое, но если ему нужно подтолкнуть — или сменился IP устройства — `/reconnect [host]` форсирует немедленную попытку подключения, независимо от того, что логгер сейчас думает о своём состоянии:

```
> /reconnect
🔌 Принудительное переподключение к устройству запущено

> /reconnect 192.168.1.50
🔌 Переподключаюсь к 192.168.1.50...
```

## Быстрый старт

**Требования:** Python 3.10+ и Meshtastic-устройство, доступное по WiFi, USB или Bluetooth LE.

```bash
git clone <этот репозиторий>
cd Mesh
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # опционально — без него подставятся разумные значения по умолчанию
```

По умолчанию клиент подключается по WiFi к `meshtastic.local`. Для USB или BLE укажите `MESH_CONN_TYPE=usb` или `MESH_CONN_TYPE=ble` в `.env` (см. [Конфигурацию](#конфигурация) ниже) — на машине с ровно одним подключённым Meshtastic-устройством этого достаточно.

Два процесса, логгер всегда запускается первым:

```bash
python3 mesh_logger.py   # фоновый демон — запустить первым, оставить работать
python3 mesh_chat.py     # интерактивный клиент — запускать и останавливать свободно
```

```bash
python3 mesh_chat.py                  # все каналы, отправка в Primary
python3 mesh_chat.py --channel ping   # только канал ping (короткая форма: --ping)
python3 mesh_chat.py --history 200    # показать больше истории при старте
```

При старте клиент показывает последние 100 сообщений (`--history` меняет это число), дальше — новые сообщения в реальном времени. Выход — `Ctrl+C` или `Ctrl+D`, логгер при этом продолжает работать.

### Запуск логгера как сервиса

Для машины, которая работает постоянно (сервер, Raspberry Pi), запустите логгер под systemd вместо терминала:

```bash
sudo cp mesh-logger.service /etc/systemd/system/
# сначала поправьте в файле WorkingDirectory / ExecStart под своё окружение
sudo systemctl daemon-reload
sudo systemctl enable --now mesh-logger
systemctl status mesh-logger
journalctl -u mesh-logger -f
```

После `git pull` `mesh_chat.py` подхватывает новый код при следующем запуске сам; логгеру нужен явный перезапуск:

```bash
git pull
sudo systemctl restart mesh-logger
```

## Команды

| Команда | Описание |
|---|---|
| `/nodes [online\|names\|hops]` | Список видимых узлов: батарея, SNR, хопы, онлайн-статус / когда был на связи |
| `/who` | Информация о себе |
| `/dm <имя> <текст>` | Личное сообщение (поиск по имени; имя с пробелами — в кавычках) |
| `/reply` | Список последних 20 сообщений, на которые можно ответить, #1 (свежее)…#20 |
| `/reply <текст>` / `/reply #N <текст>` | Ответ с настоящей цитатой — на последнее сообщение или на #N |
| `/react <эмодзи>` / `/react #N <эмодзи>` | Отправить тапбэк-реакцию |
| `/ch [имя\|all]` | Показать каналы или переключить сессию на один (или обратно на все) |
| `/send <канал> <текст>` | Разовая отправка в канал без переключения сессии |
| `/search <текст>` | Полнотекстовый поиск по всей истории логов |
| `/last <имя>` | Последние сообщения одного отправителя (точное совпадение имени) |
| `/stats [день\|узел]` | Статистика сообщений: общая, по дням или по самым активным |
| `/trace <имя>` | Traceroute — SNR каждого хопа туда и обратно |
| `/ping <имя>` | Время отклика (RTT) и число хопов до узла |
| `/pos <имя>` | Позиция узла, расстояние и азимут от вас |
| `/mute <имя>` / `/unmute [имя]` | Скрыть отправителя из живой ленты (поиск всё равно находит) |
| `/updatenames` | Запустить внеочередной обход резолвинга имён через OneMesh |
| `/settings [параметр значение]` | Посмотреть или изменить настройки локального узла — см. [`SETTINGS.ru.md`](SETTINGS.ru.md) |
| `/botping 0\|1` | Включить/выключить бота-автоответчика |
| `/autoping [минуты\|off\|text ...]` | Настроить канарейку связи: частоту, свой текст или выключить |
| `/reboot` / `/reboot confirm` | Перезагрузить узел (двухшаговое подтверждение) |
| `/reconnect [host]` | Заставить логгер немедленно переподключиться к устройству, опционально по новому адресу (только WiFi) |
| `/clear` | Очистить экран |
| `/help` | Справка по командам |

`Tab` дополняет команды, имена узлов и названия каналов.

## Конфигурация

Все настройки — в `.env` (скопируйте из `.env.example`); каждый ключ опционален и имеет разумное значение по умолчанию.

| Ключ | По умолчанию | Назначение |
|---|---|---|
| `MESH_CONN_TYPE` | `wifi` | `wifi` \| `usb` \| `ble` — как логгер подключается к устройству |
| `MESH_LANG` | `ru` | `ru` \| `en` — язык интерфейса (сообщения, ответы бота, описания настроек) |
| `MESH_HOST` | `meshtastic.local` | Адрес устройства для WiFi |
| `MESH_USB_PORT` | *(автоопределение)* | Serial-порт для USB, например `/dev/ttyUSB0` |
| `MESH_BLE_ADDRESS` | *(автоопределение)* | MAC/UUID для BLE |
| `PING_CHANNEL` | `Ping` | Имя канала для ping-бота и автопинг-канарейки |
| `BOTPING_ENABLED` | `0` | Управляется командой `/botping 0\|1` — вручную менять не нужно |
| `AUTOPING_INTERVAL_MIN` | `120` | Управляется командой `/autoping` — минуты между сообщениями канарейки, `0` выключает |
| `AUTOPING_TEXT` | *(пусто = дефолт)* | Управляется командой `/autoping text ...` — свой текст канарейки |
| `NODE_LONG_NAME` / `NODE_SHORT_NAME` / `NODE_ID` | *(пусто)* | Заполняется само командой `/who` — вручную менять не нужно |

## Под капотом

| Компонент | Технология | Роль |
|---|---|---|
| Язык | Python 3.10+, `asyncio` | Оба процесса |

| IPC | Unix-сокет, newline-delimited JSON | Связь логгер ⇄ клиент |

## Структура проекта

```
mesh_common.py        общие константы и формат/парсер строк лога
mesh_logger.py         фоновый демон: соединение с устройством + логи + IPC-сервер
mesh_chat.py           интерактивный клиент: IPC-сокет + история + TUI
test_mesh_common.py    тесты формата/парсера строк лога
mesh-logger.service    пример systemd-юнита для mesh_logger.py
SETTINGS.ru.md         полный справочник параметров /settings (англ. — SETTINGS.md)
node_names_cache.json  состояние /mute
onemesh_cache.json     имена узлов, резолвленные через OneMesh (пишет mesh_logger.py)
logs/                  логи чата по дням, chat-YYYY-MM-DD.log
```
